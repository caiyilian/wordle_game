## 1. 现状问题分析

### 1.1 代码质量问题

| 问题 | 位置 | 严重程度 |
|------|------|----------|
| **游戏逻辑重复** — `game.py` 的 `Wordle.guess()` 和 `app.py` 的 `GameSession.guess()` 实现了几乎相同的绿/黄/灰颜色判定逻辑，两份代码需同步维护 | `game.py:79-90` / `app.py:26-60` | 🔴 高 |
| **全局可变状态** — `global_room` 字典作为单例存储，Python 多线程下不安全，重启即丢失 | `app.py:9-12` | 🔴 高 |
| **硬编码词库** — Web 版始终使用 CET4，`game.py` 的 `dic_list` 功能在 Web 版中未暴露 | `app.py:17` | 🟡 中 |
| **Debug 模式暴露** — `app.run(debug=True)` 生产环境有安全风险 | `app.py:212` | 🔴 高 |
| **魔法字符串** — 颜色 `"green"/"yellow"/"gray"` 散落在后端和前端 | 多处 | 🟡 中 |
| **职责混合** — `app.py` 同时负责路由、游戏逻辑、状态管理、聊天记录 | `app.py` 全文件 | 🔴 高 |
| **控制台版与 Web 版不一致** — `play_console.py` 走图片渲染，Web 版走 DOM 渲染，两套输出 | `play_console.py` | 🟡 中 |
| **提示需手动刷新** — 发送 `提示` 命令才能更新线索，不会自动展示新信息 | `app.py:160-164` | 🟡 中 |

### 1.2 功能缺失

| 功能 | 期望 | 优先级 |
|------|------|--------|
| **多人联机** | 创建/加入房间，多玩家轮流或同时猜测同一单词 | P0 |
| **词库选择** | 游戏开始前可选 CET4/6/考研/TOEFL/GRE 等 | P1 |
| **实时更新** | WebSocket 推送，无需手动刷新或轮询 | P0 |
| **用户身份** | 昵称、头像、分数、段位 | P1 |
| **游戏统计** | 胜率、连胜、猜词分布、历史记录 | P2 |
| **虚拟键盘** | 屏幕上点击字母输入，减少打字操作 | P1 |
| **动画效果** | 卡片翻转、颜色显现、胜利庆祝 | P1 |
| **暗色模式** | 跟随系统或手动切换 | P2 |
| **国际化** | 至少中/英文界面 | P2 |

### 1.3 UI 问题

- **没有虚拟键盘** — 完全依赖物理键盘输入，移动端体验差
- **无动画** — 卡片没有翻转效果，视觉反馈平淡
- **配色老旧** — 使用系统默认 UI 色，没有设计感
- **无响应式布局** — 在手机上布局会被挤压
- **聊天交互怪异** — 通过打字输入 `猜单词` 指令而非按钮点击，不符合大众用户习惯（原版 Wordle 是点击键盘）
- **无加载/空状态** — 没有游戏状态提示，新手不知道如何开始

## 2. 技术选型决策

### 2.1 总体选型结论

```
后端：   FastAPI (Python) — 替代 Flask
API：    REST + WebSocket
数据库： PostgreSQL (云服务器) / SQLite (开发)
前端：   Vue 3 + Vite + TypeScript + Tailwind CSS
桌面端： Electron + electron-builder (Vue 3 套壳，输出 exe/dmg)
小程序： uni-app (Vue 3 驱动，编译到微信小程序)
实时：   WebSocket (FastAPI 原生支持)
```

### 2.2 为什么是 FastAPI 而不是 Flask？

| 对比维度 | Flask（当前） | FastAPI（推荐） |
|----------|--------------|-----------------|
| WebSocket | 需 Flask-SocketIO（额外依赖，不够原生） | 原生 ASGI 支持 WebSocket |
| 异步能力 | 同步，性能瓶颈 | 原生 async/await，高并发 |
| 数据校验 | 手动 `request.json` + 手写校验 | Pydantic 模型自动校验和文档 |
| API 文档 | 无自动生成 | 自动 OpenAPI/Swagger UI |
| 类型提示 | 无强制 | 原生支持，编辑器体验好 |
| 社区趋势 | 稳定但过时 | 活跃，Python 异步生态主流 |

### 2.3 为什么是 Vue 3 而非 React？

| 对比维度 | React | Vue 3（推荐） |
|----------|-------|--------------|
| 小程序生态 | Taro（第三方案，Bug 较多） | uni-app（官方维护，成熟稳定） |
| 学习曲线 | JSX + Hooks 心智模型较重 | 单文件组件 + Composition API，更直观 |
| 本项目适配 | 需要额外搭配路由/状态管理 | 内置过渡动画、响应式系统开箱即用 |
| 跨平台 | React Native（但不用来写 Web） | uni-app：一套代码 = Web + 小程序 |
| 包体积 | 较大（~40KB gzip） | 较小（~20KB gzip） |

**核心决策理由**：你同时需要 **Web 版** 和 **微信小程序版**。Vue 3 + uni-app 可以用同一套 Vue 组件代码同时编译到 Web 和微信小程序，维护成本减半。如果选 React，则 Web 和 小程序需要用不同的框架（React + Taro），生态和稳定性都弱于 uni-app。

### 2.4 为什么 Tailwind CSS？

- **快速构建 UI**——不写自定义 CSS，类名即样式
- **设计一致性**——内置设计系统（间距、颜色、断点）
- **暗色模式**——`dark:` 前缀一行搞定
- **Tree-shaking**——生产环境仅保留用到的类，体积极小

### 2.5 Electron 桌面端方案

**思路**：Electron 本质上就是一个「浏览器壳子」。我们的 Vue 3 前端已经是一个完整的 SPA，Electron 只需将其加载为原生窗口——**几乎零代码改动**。

**具体做法**（基于 `vite-plugin-electron`）：
1. 在前端项目根目录安装 `electron` + `vite-plugin-electron` + `electron-builder`
2. 写一个 `electron/main.ts`（浏览器窗口创建 + 加载 `localhost:5173` 开发或 `dist/index.html` 生产）
3. `vite.config.ts` 加一行 `electron()` 插件
4. `npm run build` 产出安装包（exe / dmg / AppImage）

**为什么这么简单**：因为真正的逻辑全在后端服务器上，桌面端只是个客户端壳子，跟浏览器完全一样——HTTP + WebSocket 连接云服务器。不需要把 Python 后端打进 exe。

**技术栈**：
| 包名 | 用途 |
|------|------|
| **electron** | 桌面框架 |
| **vite-plugin-electron** | Vite 集成，开发热更新免重启 |
| **electron-builder** | 打包为 exe/dmg/AppImage |

> **Electron 臃肿？** 用户说不追求轻量化，那 Electron 就是最省力的桌面方案。一个项目同时覆盖 Web + 桌面 + 小程序，三端复用同一套 Vue 代码。

---

#### ❓ 常见疑问：Electron 桌面端能和网页端、微信小程序联机吗？

**能。完全互通，没有任何障碍。** 原因如下：

```
Electron 桌面端  =  一个固定尺寸的 Chrome 浏览器窗口
                  （只是去掉了地址栏和标签页）
```

从后端服务器（FastAPI）的视角来看，以下三个客户端**完全等价**：

| 客户端 | 本质 | 连接方式 | 协议 |
|--------|------|----------|------|
| 浏览器 | Chrome/Firefox/Safari 标签页 | HTTP + WebSocket | 跟服务器完全一样 |
| Electron | Chromium 内核的无框窗口 | HTTP + WebSocket | **同上** |
| 微信小程序 | 微信内置 WebView | HTTPS + WSS | **同上** |

**它们都调用同一套 API**（`POST /api/rooms`、`GET /api/games/{id}`），连接**同一个 WebSocket 房间**（`ws://host/ws/{room_id}`），发送和接收**完全相同的 JSON 数据格式**。

所以在一局游戏里，可以是这样的组合：

```
  房间 "ABC123"
  ├── 玩家 A — Windows 电脑 Electron exe
  ├── 玩家 B — Mac 浏览器 Safari
  ├── 玩家 C — 手机微信小程序
  └── 玩家 D — iPhone 浏览器
```

四人同时猜同一个单词，WebSocket 广播让所有人实时看到每步猜测。服务器不关心、也**无法区分**谁是什么端。

**唯一的注意事项**：Electron 打包后访问的是线上服务器地址（`https://your-domain.com`），所以只要你的手机能访问同一个公网地址，就完全没问题。

### 3.2 核心数据流

```
用户 A 猜词：
  ① 前端点击字母 → 拼成单词 → 点击提交
  ② WebSocket 发送至 FastAPI 服务
  ③ 服务端验证单词合法性 + 计算颜色矩阵
  ④ 存入数据库 (游戏记录表)
  ⑤ 通过 WebSocket 广播给房间内所有用户
  ⑥ 所有客户端实时更新游戏面板 + 反馈动画

聊天消息：
  ① 任意用户发送消息
  ② WebSocket → 服务端 → 广播给房间所有人
```

### 4.2 核心游戏逻辑重构 (core/wordle.py)

将 `game.py` 的 `Wordle` 类和 `app.py` 的 `GameSession` 类合并为一个**纯函数式无状态模块**：

```python
# 核心：只做计算，不存状态
def evaluate_guess(answer: str, guess: str) -> list[ColorResult]:
    """输入正确答案和猜测，返回每个字母的颜色矩阵"""

def is_valid_word(word: str) -> bool:
    """检查单词是否合法（复用 pyspellchecker）"""

def get_available_word_lengths() -> list[int]:
    """返回当前词库支持的所有单词长度"""

def get_random_word(word_bank: str, length: int) -> tuple[str, str]:
    """从指定词库随机选词，返回 (单词, 释义)"""
```

这样做的好处：
- 游戏逻辑可以**独立单元测试**
- Web 版和未来可能的机器人/AI 对手可复用同一套逻辑
- FastAPI WebSocket handler 直接调用，无需实例化 GameSession

### 4.6 词库加载优化

当前词库 JSON 结构良好（`word: {中释, 英释}`），但加载方式是**每次读取整个文件+实时过滤**。优化方案：

- **启动时预加载**——将所有词库按长度分组缓存到内存字典 `dict[word_bank][length] -> list[(word, meaning)]`
- **内存占用**：10 个词库文件，总共约 ~7MB，内存中存为 Python 对象约 ~50MB，完全可接受
- 或者使用 **Redis** 存储预处理后的词库，实现多实例共享

## 6. 多人联机方案

### 6.1 游戏模式

**模式一：轮流猜词（推荐首发）**

- 房间内所有玩家共用一个单词
- 每局游戏每人轮流猜测（可用键盘/点击输入）
- 每个人看到所有人的猜测和颜色反馈
- 谁先猜出谁赢
- 如果都没猜出，按接近程度或平局

**模式二：各自独立（未来扩展）**

- 每个玩家独立猜同一单词
- 互不可见对方的猜测
- 猜完进行结果对比（类似 Wordle 分享图）

**模式三：多人竞技（未来扩展）**

- 限时竞速：所有玩家同时开始，看谁先猜出
- 轮次积分：多轮后累计得分

### 6.2 WebSocket 房间管理

```
创建房间:
  ① 用户 A POST /api/rooms → 返回 room_id + 短码
  ② 用户 A 通过 WebSocket 连接 ws://host/ws/{room_id}?user_id=xxx
  ③ 用户 B 知道 room_id → 同样连接 WebSocket
  ④ 服务端广播 player_joined

游戏流程:
  ① 房主发送 {"type": "start_game"}
  ② 服务端选词 → 广播 game_start 事件（含 word_length, max_guesses，不含答案）
  ③ 任意用户发送 {"type": "guess", "word": "apple"}
  ④ 服务端校验 → 计算颜色 → 广播 guess_result
  ⑤ 如果猜中 → 广播 game_over (含答案、释义、赢家)
  ⑥ 如果次数用完 → 广播 game_over (含答案，无人获胜)
```

### 6.3 服务端 WebSocket Manager

```python

    """管理 WebSocket 连接"""

    async def disconnect(self, room_id: str, user_id: str)
    async def broadcast(self, room_id: str, event: dict, exclude: str = None)
    async def send_to(self, user_id: str, event: dict)
```

## 9. 额外功能创意

> 除了「现有功能补齐」和「方案内功能」，以下是不错的扩展点，按「爽度/性价比」排序。

### P0 · 做了就很爽

**① 每日挑战 (Daily Challenge)**

像原版 Wordle 一样，每天所有用户同一个单词，猜完生成分享图。这是最强的**拉新/留存**功能：
- 每日 0 点自动换词（apscheduler 定时任务）
- 生成 🟩🟨⬜ 方块图分享到微信/朋友圈
- 月度/年度猜词日历（哪天猜了、结果如何）
- 后端存一份 daily_words 表，每日从词库随机选一个

**② 结果分享图生成**

游戏结束后生成一张精美的结果图（带 🟩🟨⬜ 方块 + 词库名 + 用时），可供分享：
- 后端用 Pillow 生成图片（已有基础设施）
- 或前端用 `html2canvas` 截图 DOM 元素
- 分享到微信聊天、朋友圈、微博

**③ 读音功能**

点击单词/字母可发音——学习类游戏的天然需求：
- 浏览器端直接用 **Web Speech API** (`speechSynthesis`) 零依赖
- 小程序端用微信 `wx.createInnerAudioContext()`
- 猜完的单词自动朗读一遍，帮助记忆

### P1 · 提升深度

**④ 成就/徽章系统**

给玩家上瘾的动力：
| 徽章 | 解锁条件 |
|------|----------|
| 🏆 初次胜利 | 第一次猜中单词 |
| 🔥 势不可挡 | 连胜 5 局 |
| 💎 完美猜词 | 一次就猜中 |
| 🧠 知识渊博 | 用过全部 10 种词库各赢一次 |
| 🎯 神射手 | 连续 3 局都在 3 步内猜中 |
| 🌙 夜猫子 | 在凌晨 0-5 点赢过一局 |
| 🤝 社交达人 | 和 10 个不同玩家联机过 |

**⑤ 排行榜**
- 全局榜：总胜率 / 连胜 / 平均猜词步数
- 词库榜：按 CET4 / GRE / 考研 分别排名
- 好友榜：只看微信好友之间的排名

- 必须使用之前揭示的黄色/绿色字母（原版 Wordle 的 Hard Mode）

### P2 · 让游戏更好玩

**⑦ AI 对手 / 单人练习模式**

- 没有联机玩家时，可以跟 Bot 对战
- Bot 策略：随机猜 / 按信息熵最优猜（参考 3Blue1Brown 的 Wordle 信息论视频）
- 对学习英语的人来说，跟 Bot 猜完还能看 Bot 的「思考过程」

**⑧ 自定义词库 / 单词本**

- 老师可以创建自定义词库（JSON 导入）
- 用户可以把自己猜错的单词加入「生词本」
- 生词本达到一定数量后自动生成一套复习题

**⑨ 限时模式 / 闪电战**

- 每步限时 30 秒，超时自动跳过
- 所有玩家同时开始，先猜出者胜
- 适合朋友聚会时玩，紧张刺激

**⑩ 押注模式**

- 每局开始前玩家可以用积分押注（猜几步内能猜中）
- 押中赢积分，积分可用于解锁特殊主题/头像框
- 增加社交博弈的乐趣

### P3 · Nice to have

**⑪ 单词详情卡片** — 猜完后展示：中文释义、英文例句、词根词缀、近义词/反义词（可选）
**⑫ 对战回放** — 游戏结束后可回放整个猜词序列，像棋谱一样复盘
**⑬ 皮肤/主题商城** — 用积分兑换不同配色主题（赛博朋克/水墨风/粉红）
**⑭ 赛季制** — 每月一个赛季，赛季末根据段位发奖励
**⑮ 微信群组** — 微信小程序内创建固定群组，群内排名，群主管理词库

## 12. 附录：决策总结

```
是否用框架？      ✅ 是 — Vue 3 (非轻量级，满足需求)
要不要 TS？       ✅ 是 — 前后端均 TypeScript
要不要数据库？    ✅ 是 — PostgreSQL
要不要 WebSocket？ ✅ 是 — FastAPI 原生 WebSocket
要不要小程序？    ✅ 是 — uni-app (Vue 生态)
要不要桌面端？    ✅ 是 — Electron 套壳，Vue 代码零改动即可输出 exe
要不要 Docker？   ❌ 否 — 直接 systemd + Nginx 部署，更简单
要不要动画？      ✅ 是 — CSS 3D 翻转 + canvas confetti
要不要保留旧代码？ ❌ 否 — 全部重写，复用词库 JSON 即可
```
